# [Home Server] 02. AWS에선 멀쩡했던 코드가 홈 서버에선 멈춘 이유

[Home Server] 02. AWS에선 멀쩡했던 코드가 홈 서버에선 멈춘 이유

홈 서버로 인프라를 옮긴 직후, 분명 잘 돌던 코드가 갑자기 동작을 멈추는 현상을 겪었다. 코드는 한 글자도 바뀌지 않았는데 환경만 바꿨더니 외부 API 호출이 매달려버리는, 처음 보는 종류의 문제였다. 결국 원인은 IPv6 였다. 이번 글에서는 그 추적 과정을 정리해두려 한다.

### 1. cloudflared로 들어오는 트래픽은 멀쩡한데, 나가는 요청이 멈춘다
홈 서버 이전 자체는 의외로 매끄러웠다. 이전 글에서 다룬 Cloudflare Tunnel 덕분에 외부에서 `dezeli.dev` 로 들어오는 요청은 깔끔하게 컨테이너까지 도달했고, 사용자 입장에서 보이는 첫 화면은 평소와 다를 게 없었다.
이상한 점은 그다음에 나타났다. 내가 서비스 하던 Mobilfit 앱에서 길찾기 기능을 누르면 응답이 없었다. 로그를 들여다보니 백엔드가 OpenRouteService, Kakao, Overpass 같은 외부 API를 호출하는 지점에서 그대로 멈춰 있었다. 에러도 안 났고, 예외도 안 던졌고, 그저 `requests.post(...)` 한 줄에서 영원히 머물러 있었다. 같은 코드가 AWS 위에선 잘 돌던 걸 분명히 봤기 때문에 더 당혹스러웠다.
처음에는 ORS API 키가 만료됐나, Cloudflare 어딘가에서 outbound도 검사하는 건가, 컨테이너 DNS가 깨진 건가 하는 식으로 의심을 옮겨다녔다. 하지만 컨테이너 안에서 `python -c "import socket; socket.create_connection(('api.openrouteservice.org', 443), timeout=20)"` 를 직접 때려보고 나서야 방향이 잡혔다. 연결 시도 자체가 매번 십수 초 동안 매달리다가 가까스로 풀렸다. DNS는 멀쩡했지만, 그 결과로 받은 주소로 TCP 연결을 거는 단계에서 시간이 통째로 사라지고 있었다.

### 2. 진짜 원인은 IPv6
원인을 좁히고 보니 어이없게도 IPv6 였다. 정확히 말하면 "IPv6가 켜져는 있는데 실제로는 안 통하는" 어정쩡한 상태가 문제였다.
컨테이너 안에서 `getaddrinfo` 가 도메인을 풀면 A 레코드(IPv4)뿐 아니라 AAAA 레코드(IPv6)도 함께 반환된다. 리눅스의 RFC 6724 기본 정책상 호스트에 사용 가능한 IPv6 source address 가 있으면 IPv6 가 우선 시도되는데, Python `requests` 가 사용하는 `socket.create_connection` 은 이 결과를 **순서대로 직렬로** 시도한다. 브라우저처럼 IPv4와 IPv6를 거의 동시에 시도하는 Happy Eyeballs(RFC 8305)는 구현되어 있지 않다. 결국 IPv6 시도가 OS 의 connect timeout 한계까지 막혀 있는 동안 프로세스는 그저 기다리는 수밖에 없다.
그렇다면 AWS 에선 왜 멀쩡했을까. 답은 단순했다. AWS VPC 는 기본이 IPv4 전용이고, 별도로 IPv6 를 활성화하지 않으면 EC2 인스턴스에도, 그 위 컨테이너에도 IPv6 주소가 붙지 않는다. IPv6 source address 가 없으니 `getaddrinfo` 는 처음부터 AAAA 레코드를 결과에서 제외하고, IPv4 로만 깔끔하게 연결이 이루어진다. "IPv6 를 잘 쓰고 있어서" 가 아니라 "IPv6 자체가 없어서" 문제가 안 났던 것이다.
반면 홈 서버는 처지가 달랐다. 한국 가정용 인터넷은 ISP 가 IPv6 를 부분적으로 배포해둔 경우가 많아, 공유기까지 IPv6 가 들어오긴 한다. 컨테이너에는 자동으로 IPv6 주소가 잡히고 AAAA 도 반환된다. 문제는 그 IPv6 패킷이 ISP 백본이나 공유기 방화벽 어딘가에서 조용히 사라진다는 점이다. 응답은 영영 오지 않고, TCP handshake 는 그저 매달려 있다. cloudflared 가 inbound 를 IPv4 로 깔끔하게 끌어오던 것과는 또 다른 이야기였다. 들어오는 트래픽과 나가는 트래픽은 완전히 별개의 경로라는 사실을 새삼 체감했다.

### 3. 컨테이너 단에서 IPv6 끄기, 그리고 타임아웃이라는 안전망
원인을 알고 나니 해결은 단순했다. 호스트나 공유기 설정까지 손대지 않고, 영향 범위를 컨테이너 안으로 한정하고 싶었기 때문에 `docker-compose.prod.yml` 의 `web` 서비스에 sysctl 두 줄을 추가했다.

```yaml
sysctls:
  - net.ipv6.conf.all.disable_ipv6=1
  - net.ipv6.conf.default.disable_ipv6=1
```

이렇게 하면 컨테이너 안에서 IPv6 자체가 꺼져버리고, `getaddrinfo` 도 더 이상 AAAA 레코드를 결과에 포함시키지 않는다. 외부 API 호출은 처음부터 IPv4 단일 경로로 진행된다. 사실상 홈 서버 환경을 컨테이너 단에서 "AWS 와 같은 IPv4 전용 상태" 로 정렬해준 셈이다.

### 4. 환경 차이가 만든 함정
이번 일을 겪으면서 인프라를 옮길 때 가장 무서운 건 "코드가 그대로인데 환경이 다르게 행동하는" 종류의 차이라는 점을 새삼 느꼈다. AWS 처럼 잘 다듬어진 환경에서는 자연스럽게 가려져 있던 가정들이, 홈 서버라는 날 것 그대로의 환경에선 하나하나 정면으로 드러난다. 그게 홈 서버의 가장 어려운 부분이자, 동시에 가장 많은 걸 배우게 해주는 부분이라는 생각이 들었다.
포트포워딩 대신 cloudflared, 안정적인 클라우드 네트워크 대신 ISP 의 어정쩡한 IPv6, 한 줄짜리 함수 호출이 만드는 거대한 영향. 이 모든 걸 이해해 가는 과정 자체가 이미 충분히 값진 경험이 되고 있다.
