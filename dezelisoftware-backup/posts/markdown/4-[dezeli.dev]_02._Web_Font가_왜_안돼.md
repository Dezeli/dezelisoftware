# [dezeli.dev] 02. Web Font가 왜 안돼?

### 로컬에서는 되는데 왜 안돼? 
윈도우 환경 기반인 로컬 PC에서는 분명히 됐는데, 이상하게 Ubuntu 22.04 기반 서버에서 배포한 dezeli.dev만 폰트가 나오지 않았다. 처음에는 서버에서만 Docker 컨테이너를 활용하기에 Docker 세팅 문제인 줄로만 알고 애먼 설정만 붙잡고 있었다. 그렇게 몇십 분을 삽질했는데, 알고 보니 원인은 허무할 정도로 단순한 곳에 있었다.

바로 리눅스의 파일 대소문자 구별 문제. 에디터에서 자동 완성해 주는 대로 타이핑을 했더니, 내가 코드에 적은 것이 폰트 파일의 실제 대소문자와 다르게 들어갔던 것이었다. 윈도우 환경에 익숙해져서 리눅스의 이 엄격함을 잠시 잊고 있었다.

```css
/* src/styles/fonts.css */
/* 실제 파일명이 MyFont.OTF라면 코드는 소문자로 적어놨었다 */

@font-face {
    font-family: 'CustomFont';
    src: url('../assets/fonts/myfont.otf') format('opentype');
}
```

### 왜 무한 로딩이지?
해결했나 싶어 한숨 돌리려던 찰나, 이번엔 사이트가 무한 로딩에 걸리며 여전히 폰트 적용이 안 됐다. 답답한 마음에 Docker 로그를 살펴보니 OTF 폰트 용량이 너무 무겁다는 경고 로그가 찍혀 있었다. 

네트워크 탭을 찍어보니 내가 쓰려던 OTF 폰트 파일 하나가 무려 17MB. 웹에서 17MB짜리 파일을 매번 다운로드하게 만들었으니 브라우저가 비명을 지를 만도 했다. 이젠 코드의 문제가 아니라 근본적인 폰트 포맷의 문제였다.

### 웹 최적화 폰트 형식  WOFF2
웹에 최적화된 형식이 절실했다. 복잡하게 생각할 것 없이 구글에 "OTF to WOFF2"를 검색해서 나오는 사이트 중 하나에 접속해 변환을 돌렸다. 결과는 놀라웠다. 17MB였던 덩치가 3MB 이하로 줄어들었다. 용량이 80% 넘게 증발했는데 웹에서 보기엔 차이가 전혀 없다.

```css
/* src/styles/fonts.css */

@font-face {
    font-family: 'CustomFont';
    src: url('../assets/fonts/myfont.woff2') format('woff2');
}
```

### 느낀 점
리눅스 서버 환경에서는 파일명 대소문자 하나도 우습게 보면 안 된다는 것, 그리고 폰트 용량을 얕잡아보면 안 된다는 것. 무엇보다 웹 폰트는 무조건 WOFF2가 답이라는 걸 뼈저리게 배웠다. 17MB짜리 무거운 짐을 덜어내고 나니 폰트 적용 속도가 눈에 띄게 빨라졌다. 다음에 폰트를 적용할 때는 확실하게 먼저 확인할 부분들이다.
